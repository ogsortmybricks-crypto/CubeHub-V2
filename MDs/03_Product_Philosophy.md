# Product Philosophy

## The Goal

The goal is not to build the biggest cubing website.

The goal is to build software that becomes indispensable.

If CubeHub disappeared tomorrow, the community should genuinely miss it.

---

## The CubeHub Test

Every feature proposal must answer four questions.

1. Does this make someone a better cuber?

2. Does this strengthen the community?

3. Does this preserve or improve cubing knowledge?

4. Will people actually use this regularly?

If the answer to all four is "no," the feature should not be built.

---

## Platform, Not Product

CubeHub should never feel like a collection of unrelated tools.

Every feature should connect to every other feature.

Learning leads to practice.

Practice generates statistics.

Statistics improve AI coaching.

AI recommends algorithms.

Algorithms belong to repositories.

Repositories create discussions.

Discussions generate research.

Research improves the algorithms.

Everything is connected.

---

## Build for Experts and Beginners

A beginner should feel welcomed.

An expert should never feel limited.

Every page should scale with experience.

---

## Simplicity Wins

Powerful software does not have to feel complicated.

Complex systems should be hidden behind intuitive interfaces.

Users should feel productive immediately.

---

## Long-Term Thinking

Do not optimize for today's trends.

Build software that can still feel modern ten years from now.

Prioritize architecture, extensibility, and maintainability over short-term convenience.
